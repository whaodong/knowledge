本轮目标：
为当前后端项目实现“基于 Spring Security + Spring Session + Redis + HttpOnly Cookie”的登录功能。
不要实现 JWT 登录，不要返回 accessToken / refreshToken。
本轮以浏览器 Web 登录为目标，采用服务端 Session 方案。

实现要求：

1. 登录方案
- 使用 Spring Security 完成用户名密码认证
- 使用 Spring Session + Redis 存储会话
- 登录成功后由服务端建立 Session
- 前端通过 Cookie 维持登录态
- 所有接口继续统一返回 knowledge.aiapp.common.result.Result<T>

2. 必须实现的接口
- POST /api/auth/login
- POST /api/auth/logout
- GET /api/auth/me

3. 接口行为要求

3.1 POST /api/auth/login
- 接收 JSON 请求体：
  {
  "username": "admin",
  "password": "123456"
  }
- 使用 AuthenticationManager 完成认证
- 认证成功后将认证信息写入 SecurityContext
- 确保会话被 Spring Session 持久化到 Redis
- 返回统一结构：
  {
  "code": 200,
  "message": "登录成功",
  "data": {
  "userId": 1,
  "username": "admin",
  "nickname": "管理员",
  "roles": ["ADMIN"]
  },
  "timestamp": "2024-04-07T17:00:00"
  }
- 禁止在 data 中返回 token
- 禁止把密码写入返回体或日志

3.2 POST /api/auth/logout
- 调用 Spring Security 标准登出流程
- 使当前 Session 失效
- 清理 SecurityContext
- 返回统一结构：
  {
  "code": 200,
  "message": "退出成功",
  "data": null,
  "timestamp": "2024-04-07T17:00:00"
  }

3.3 GET /api/auth/me
- 从当前登录上下文中获取用户信息
- 若已登录，返回当前用户基础信息和角色
- 若未登录，返回统一未授权结构

5. 安全与配置要求
- 使用 BCryptPasswordEncoder
- 登录、登出、获取当前用户接口补充 OpenAPI 注解
- 对 /api/auth/login 允许匿名访问
- 对 /api/auth/logout、/api/auth/me 要求已登录
- 除登录与健康检查等白名单接口外，其余接口默认需要认证
- 保留 CSRF 保护，不要为了省事全局关闭；如果当前前后端交互需要特定处理，请按浏览器 Cookie 场景给出合理实现
- 配置 CORS，允许前端携带凭证请求
- Session Cookie 需要有清晰配置项，至少考虑 HttpOnly、Secure、SameSite
- 敏感配置通过配置文件占位或环境变量读取，禁止硬编码
